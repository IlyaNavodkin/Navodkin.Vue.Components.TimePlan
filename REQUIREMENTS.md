# Требования к новому компоненту командного планировщика

## 1. Иерархия строк и сворачивание
- Дерево строк должно поддерживать вложенность: `Проект -> Чардж -> Сотрудник`.
- Один проект содержит много чарджей.
- Один чардж содержит много сотрудников.
- Строки уровня `Проект` и `Чардж` являются групповыми и должны сворачиваться/разворачиваться.
- При сворачивании группы скрываются все дочерние строки.

## 2. Выделение диапазона дат и создание таймлайн-блока
- На каждой строке должна быть доступна работа с диапазоном дат по ячейкам дня.
- При выделении диапазона подсвечиваются:
  - выбранные ячейки строки,
  - заголовки дней (верхняя шкала дат) для выбранного диапазона.
- Через контекстное меню на строке сотрудника должна быть команда создания блока таймлайна из выделенного диапазона.
- Созданный блок должен быть связан с конкретным сотрудником.

## 3. Горизонтальная навигация и масштаб
- Весь график должен строиться на диапазоне одного календарного года.
- Над графиком должен быть выбор года.
- Горизонтальный скролл должен перемещать пользователя внутри выбранного года.
- Масштабирование должно менять ширину видимой области/детализацию, но не менять полный годовой диапазон данных.
- Должны быть фиксированные уровни масштабирования:
  - `1 неделя`,
  - `3 недели`,
  - `1 месяц`,
  - `3 месяца`.
- В масштабе `3 месяца` не нужно показывать отдельные дни и дневные ячейки; вместо них показываются названия месяцев и месячная сетка.
- Переключение масштаба не должно ломать текущую структуру данных, строки и существующие таймлайн-бары.
- Выходные дни должны визуально отмечаться отдельным цветом в заголовках и ячейках сетки.

## 4. Управление таймлайн-барами сотрудников
- Таймлайн-бары сотрудников должны поддерживать:
  - перетаскивание по времени,
  - изменение длительности (resize за левый/правый край),
  - удаление.
- Через контекстное меню по бару обязательно 2 пункта:
  - `Редактировать`,
  - `Удалить`.
- Если в чардже не создано ни одного бара у сотрудников, чардж должен оставаться корректно отображаемым без ошибок.
- Таймлайн-бары одного сотрудника не должны накладываться друг на друга.
- При создании, редактировании, перетаскивании или изменении размера бара система должна запрещать пересечение с другими барами этого сотрудника.
- Если изменяемый или создаваемый бар пересекается с существующим баром, операция не должна применяться и пользователь должен получить уведомление.

## 5. Совместимость левой и правой частей таблицы
- Левая фиксированная часть (названия проекта/чарджа/сотрудника и управление сворачиванием) не должна конфликтовать с правой временной частью.
- При горизонтальном скролле левая колонка остается видимой и выровненной по строкам с правой частью.
- Заголовки дат и строки не должны перекрываться некорректно (z-index/слои должны быть согласованы).

## 6. Уведомления (тосты)
- Должны быть тост-уведомления при операциях:
  - сохранение,
  - редактирование,
  - удаление.
- Тосты показываются без перезагрузки страницы и автоматически скрываются через короткое время.

## 7. Автодобавление сотрудника в чардж при создании бара
- Строка чарджа не используется для создания таймлайн-бара.
- Через контекстное меню левой sticky-ячейки строки чарджа (там, где название чарджа) должна быть доступна команда `Добавить сотрудника`.
- После добавления сотрудника в чардже появляется новая строка сотрудника.
- Через контекстное меню левой sticky-ячейки строки сотрудника (там, где имя сотрудника) должна быть доступна команда `Удалить сотрудника`.
- Контекстное меню на правой таймлайн-части строки сотрудника используется для действий с таймлайн-блоками, а не для удаления сотрудника.
- При удалении сотрудника удаляется его строка и связанные с ним таймлайн-бары.
- Строка чарджа является read-only для таймлайн-баров и показывает только агрегированную занятость сотрудников в виде линии/полос занятости.

## 8. Техническая архитектура и декомпозиция
- Код компонента должен быть максимально декомпозирован.
- Бизнес-логика, состояние, вычисления дат, зум, выделение диапазона, drag/resize и управление барами должны быть вынесены в composables.
- Composables могут использовать другие composables меньшего уровня.
- Composables нижнего уровня также могут быть декомпозированы и использовать еще более мелкие composables, если это снижает сложность.
- UI должен быть разбит на компоненты.
- Компоненты могут использовать внутри себя другие компоненты.
- Компоненты не должны содержать крупную бизнес-логику; они должны получать состояние/действия через props, events или composables.
- Целевая структура не должна быть монолитной реализацией в одном файле.
